# 可靠性基线

复杂 ToB 系统的可靠性不能等到上线前再补，它必须从骨架阶段就开始约束。

## 当前基线

- 所有非平凡改动都应有明确验证计划。
- 验证优先从最小相关范围开始，再扩展到更广范围。
- 涉及稳定逻辑的改动，默认应补或更新单元测试，作为最小回归护栏之一。
- 执行计划里必须写出“验证了什么”和“还没验证什么”。
- 功能清单变化应与验证记录保持对齐，避免能力状态和验证状态脱节。
- 运行、发布、排障知识最终要沉淀到 `.ch/docs/runbooks/`。

## 未来必须建设的能力

- 本地可启动、可复现、可观察
- 健康检查、就绪检查、关键依赖探活
- 关键路径自动化验证
- 可持续维护的单元测试基线
- 日志、指标、链路的统一入口
- 性能预算和回归阈值

## 对代理最重要的可靠性要求

- 能自己发现失败
- 能自己复现失败
- 能自己读取足够的运行上下文
- 能在计划中留下明确的失败点与后续动作

## runbook 入口

- 运行手册统一收口到 `.ch/docs/runbooks/README.md`
