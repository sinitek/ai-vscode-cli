# 功能清单治理

功能清单不是“上线前随手写一版”的交付物，而是需求、实现、测试、收尾之间的对齐层。

当前仓库把功能清单拆成两个层次：

- **总表**：`.ch/docs/product-specs/FEATURE_INVENTORY.md`
- **单项规格**：`.ch/docs/product-specs/*.md`

总表回答“现在系统里到底有哪些能力、状态如何”；单项规格回答“某个能力为什么存在、范围是什么、如何验收”。

## 单一事实来源

- 所有仓库级或业务域功能，都应最终在 `.ch/docs/product-specs/FEATURE_INVENTORY.md` 留下一条记录。
- 不要让 README、聊天记录、任务单、临时表格分别维护不同版本的功能列表。

## 什么时候必须更新功能清单

- 新增一个用户可感知能力、后台能力、运维能力或平台能力。
- 修改已有能力的行为、权限、流程、入口、状态或适用角色。
- 一个能力进入暂停、废弃、替换、下线状态。
- 一个需求虽然没有新增页面，但改变了导入导出、通知、审计、报表或批处理能力。

## 总表建议字段

- 业务域
- 功能名称
- 状态
- 主要角色
- 规格来源
- 实现位置
- 测试状态
- 备注

其中“测试状态”不是为了重复测试报告，而是为了快速回答：这项能力当前是否有基础自动化护栏。

## 状态建议

- `proposed`：已识别，但尚未进入实施
- `in-progress`：正在建设
- `active`：已经交付且当前生效
- `deprecated`：仍存在，但不建议继续使用
- `removed`：已下线，仅保留历史记录

## 与计划和规格的联动

- 执行计划应明确：本次是否需要更新功能清单。
- 产品规格应标注：完成后要在总表新增还是更新哪一项。
- 收尾说明应记录：本次实际变更了哪些功能项。

## 当前骨架阶段的约束

- 仓库还没有真实业务域，因此允许先记录仓库级 harness 能力。
- 一旦出现真实业务代码，功能清单应优先按业务域组织，而不是继续只记“基础设施能力”。
- 如果一个功能没有规格来源，至少要能追溯到执行计划或设计文档。

## 明确禁止

- 不要把“页面名”当成功能名。
- 不要把“接口名”直接当成用户能力描述。
- 不要只在需求开始时更新一次，收尾时却忘记同步最终状态。
