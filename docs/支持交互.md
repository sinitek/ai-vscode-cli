# 支持交互

## 背景

目前codex,claude交互都是在点击发送后创建一个进程执行命令，这种方式存在一些问题：
- 进程资源没有复用
- 无法压缩上下文

## 设计

当前实现采用两条交互路径：
- Codex：直接调用用户本机安装的官方 `codex` CLI，通过 `codex app-server --listen stdio://` 建立 JSON-RPC 会话，先完成 `initialize` / `initialized` 握手；其中 `clientInfo` 统一声明为 `codex` 并优先探测本机已安装 Codex 版本，再通过 `thread/start` / `thread/resume` + `turn/start` 做会话续接。
- Claude：继续使用 `claude-agent-sdk` 实现交互组件，以复用进程资源并压缩上下文。

## 当前实现梳理（基于代码）

- 入口：`src/extension.ts` 的 `runPrompt()` 会按 CLI 分流；Codex/Claude 走交互 runner，Gemini 继续调用 `runCliStream()` 创建一次性进程。
- 会话：扩展侧维护 session 列表；Codex 侧通过 threadId 做会话映射，并在每轮任务里用 `thread/start` / `thread/resume` 延续历史；Claude 继续复用 SDK session。
  - codex：`codex app-server --listen stdio://` + JSON-RPC（`thread/start|resume`、`turn/start`）
  - claude：`claude --print --output-format stream-json --include-partial-messages --resume <sessionId> <prompt>`
  - gemini：`gemini --resume <sessionId> <prompt>`（本次需求暂未提及是否纳入交互模式）
- 流式：Claude 走 stream-json 并解析 tool 事件；Codex 走 app-server JSON-RPC 事件流（assistant delta / plan / item started-completed）；Gemini 继续使用一次性 stdout/stderr 流。

## 目标（待确认）

- 复用进程资源：同一会话内多轮对话不重复冷启动 CLI/Agent。
- 上下文压缩：在长会话中控制 token/上下文体积，避免性能下降与模型退化。
- 兼容现有能力：保留现有 UI/会话列表/日志/停止运行/思考档位等能力，必要时可降级回“一问一进程”。

## 方案选项（待你选择）

### 方案 A：接入 SDK（历史方案）

- Codex：历史上接入过 `@openai/codex-sdk`（SDK 会 spawn `codex` 并用 JSONL 事件与之交互；同一 Thread 可多次 `run()`）。
- Claude：接入 `@anthropic-ai/claude-agent-sdk`（推测提供类似“常驻 Agent/会话 + 流式事件”的抽象）。
- 优点：交互协议/事件类型更稳定，便于做流式 UI、工具事件、线程恢复；更符合“支持交互”的语义。
- 风险：依赖体积与打包；Claude SDK 细节需验证（会话/线程持久化、事件格式、是否也依赖本地 claude-code）。

### 方案 B：不接 SDK，自建常驻进程协议

- 自己启动 CLI 的交互模式（如果 CLI 支持），通过 stdin 写入请求、stdout 解析响应。
- 优点：依赖少、可控；缺点：协议不稳定/跨平台坑多，维护成本高。

### 方案 C：只做“上下文压缩”，不做进程复用

- 继续“一问一进程”，但在发送前自动摘要历史，把摘要作为“系统/前缀”注入，或新建会话从摘要继续。
- 优点：改动小；缺点：仍有冷启动成本，且 CLI 自身的会话/线程与扩展侧摘要如何对齐需要设计。

## 上下文压缩（需要产品决策）

- 触发方式：
  - 自动：达到阈值（轮数/字符数/估算 token）后触发
  - 手动：用户点“压缩上下文”按钮
  - 混合：自动提示 + 用户确认
- 压缩策略候选：
  - 保留最近 N 轮原文 + 更早内容写成“会话摘要”
  - 维护“事实记忆（facts）/待办（todos）/决策（decisions）/约束（constraints）”四段结构
  - 压缩后是否新开线程（旧 thread 冻结）vs 在同一 thread 继续（若 SDK 支持）
- 可见性：
  - 是否把摘要作为一条 trace 消息展示（默认建议展示，便于用户信任与可回滚）

## 会话与线程映射（需要明确）

- 交互模式下：
  - 扩展 sessionId ⇄ SDK threadId / agent sessionId 的映射如何存储（建议存到 `~/.sinitek_cli/sessions` 的记录里）。
  - “切换会话/重启 VS Code/扩展崩溃”后如何恢复：`resumeThread(threadId)` 还是重新 spawn 并恢复。

## 未明确问题（本轮先记录，后续逐条确认）

- 范围：是否只做 codex+claude，还是也要把 gemini 纳入“交互模式”？
- UI：是否要在面板上提供“交互模式（开/关）”切换？默认开还是默认关？
- 资源生命周期：常驻进程的释放策略（空闲超时/切换会话即销毁/手动关闭/VS Code 退出即销毁）。
- 压缩触发：阈值的默认值与可配置项（轮数/字符/token），以及是否需要“压缩前提示”。
- 失败与降级：SDK 初始化失败、进程崩溃、恢复失败时的用户体验（自动降级到“一问一进程”？是否提示原因？）。
- 文档/链接：本仓库当前未配置 `chrome-devtools-mcp`，暂无法用 MCP 打开链接；可用 HTTP 抓取页面内容辅助设计与实现验证。

## 决策记录（已确认）

- 范围：仅 codex + claude 纳入交互模式（gemini 暂不纳入）【1A】
- 技术路线：Claude 继续走 SDK；Codex 改为直接调用官方 CLI 并通过 `resume <threadId>` 续接会话【更新】
- 上下文压缩：混合模式（达到阈值先提示，用户确认后压缩）【3B】
- 资源生命周期：每个 CLI 仅维护 1 个“当前会话”的交互状态；切换会话释放旧状态；需要时基于 threadId/sessionId 恢复【4A，按现实现修正】
- 交互模式开关：提供开关，默认开启（可标注 Beta）【1B】
- 压缩阈值：综合阈值（任一满足触发；先实现轮数 + 字符数）【2D】
- 摘要结构与继续方式：摘要为“4 段 + 关键文件/命令/结论索引”；压缩后新开 thread 继续、旧 thread 冻结【3C+2】
- 交互模式开关入口与作用域：入口采用“设置项 + 面板临时开关”（面板覆盖设置）；作用域按“工作区级”【建议落地：MVP 先做设置项，后补面板开关】
- 交互模式开关的落地解释：面板开关本质写入“工作区设置”；工作区设置覆盖全局默认值（不做纯内存的临时覆盖，避免用户困惑与不可追溯）
- 压缩提示交互：发送前提示；使用 VS Code 弹窗按钮；提供“这次跳过 + 本会话不再提示”【A + A + (A+B)】
- 压缩后的可见性与回滚：system + trace 同时展示摘要；MVP 不做回滚按钮，但保存冻结 threadId 以便后续恢复【C + A】
- 压缩阈值默认值：轮数 30 或累计字符 24000（满足任一则发送前提示压缩）；不提供“本工作区不再提示”按钮
- SDK metadata：与 sessions 同目录新增 `~/.sinitek_cli/sessions/<workspaceKey>.meta.json`（不改 messages 文件结构）
- 摘要任务执行：使用当前 CLI（codex 会话由 codex 生成摘要；claude 会话由 claude 生成摘要）
- Runner 生命周期：24 小时空闲自动释放；冻结 thread 最多保留最近 5 个
- 压缩后新 thread 启动上下文：摘要 + 最近 3 轮原文
- 停止运行语义：停止当前 turn 并重建 runner（保留 threadId，下次自动 resume）

## MVP 验收标准（建议）

- 同一会话内连续发送 3 次，codex/claude 不重复冷启动（可通过 pid/runnerId/日志验证）。
- 切换到另一会话时，旧会话 runner 释放；切回旧会话可恢复并继续对话（失败则自动降级为“一问一进程”，且有可理解的提示）。
- 达到压缩阈值时出现提示；用户确认后生成“会话摘要”并继续对话，且用户能看见摘要内容（trace/system 形式均可）。
- 停止运行：能终止当前 turn（并尽可能保留已产生的输出）；runner 后续仍可继续使用或按策略重建。
- 不破坏现有气泡规则：消息按时间顺序展示；只允许追加到“上一条同类型气泡”，禁止回填到更早气泡。

## 设计细节待确认（下一轮你选）

### 交互模式开关的呈现与作用域

现状约束：Webview 目前只有 CLI 选择、思考档位、会话管理、规则编辑等；没有“按钮型交互卡片”组件，最省改动的是走 VS Code 的 `showInformationMessage/showWarningMessage` 或在设置里配置。

- 开关入口：
  - A 仅设置项（Settings）：`sinitek-cli-tools.interactive.codex/claude`（默认 true）
  - B 仅面板开关（Webview）：下拉/checkbox（默认 true）
  - C 两者都有：面板是临时开关（覆盖设置项，仅本次会话/本工作区有效）
  - 推荐：C（落地顺序：MVP 先 A，后补 C）
- 作用域：
  - A 全局（用户级）
  - B 工作区级
  - C 会话级（每个 session 可独立开关）
  - 推荐：B（避免不同项目习惯互相影响；实现复杂度中等）

### 压缩提示的交互方式（混合模式）

- 提示时机：
  - A 发送前提示（点击发送后，如果触发阈值，先弹确认，再真正发给模型）
  - B 发送后提示（本轮先正常发，结束后提示“建议压缩”）
  - 推荐：A（避免继续膨胀；但会增加一次确认）
- 提示位置：
  - A VS Code 弹窗（Information/WarningMessage）带按钮：`现在压缩 / 这次跳过 / 本会话不再提示`
  - B Webview 系统气泡提示 + 链接（需要额外实现点击处理）
  - C Webview 顶部 banner（需要额外 UI）
  - 推荐：A（最快落地，且符合“需要确认”）
- 跳过策略：
  - A 只跳过 1 次
  - B 本会话不再提示（直到手动压缩/新会话）
  - C 本工作区不再提示（可在设置恢复）
  - 推荐：A + B（按钮同时提供）
  - 落地解释：
    - “本会话不再提示”默认只在扩展内存中生效（不落盘）；新建会话后恢复提示

### 压缩阈值默认值（已确认）

默认值（可后续改为可配置）：

- 轮数阈值：最近累计对话达到 30 轮后开始提示
- 字符阈值：会话消息累计字符达到 24000 后开始提示
- 触发逻辑：满足任一阈值 -> 发送前弹窗提示压缩
- 弹窗按钮：不提供“本工作区不再提示”

### SDK 会话映射与落盘（已确认）

需要把扩展侧 sessionId 与 SDK 的 threadId / agent sessionId 做映射，否则无法在“切换会话/重启 VS Code”后恢复交互模式。

- 建议落盘方案（MVP）：
  - 新增 metadata 文件：`~/.sinitek_cli/sessions/<workspaceKey>.meta.json`
  - 结构（示例）：
    - `byCli.codex[sessionId] -> { threadId, frozenThreadIds: string[], updatedAt }`
    - `byCli.claude[sessionId] -> { agentIdOrSessionId, frozenIds: string[], updatedAt }`
  - 原则：不改动现有 messages JSON 结构；metadata 与消息分离，避免历史兼容风险。
落盘位置：metadata 放在 sessions 文件同目录（同 workspaceKey）

### 压缩执行流程（建议）

目标：压缩后新开 thread 继续、旧 thread 冻结；同时满足“消息气泡规则”与“可见性”。

- 建议流程：
  - 1 触发阈值 -> 发送前弹窗：`现在压缩 / 这次跳过 / 本会话不再提示`
  - 2 用户点“现在压缩”：
    - 2.1 在旧 thread 上跑一次“摘要任务”（使用固定结构 C：4 段 + 索引）
    - 2.2 将摘要写入消息流：先 system（短摘要/提示），再 trace（完整摘要）
    - 2.3 冻结旧 threadId：写入 metadata 的 frozenThreadIds
    - 2.4 新开 thread：把摘要作为“启动上下文”（例如作为第一条 prompt 的前置块）并继续执行用户原始 prompt
  - 3 如果摘要任务失败：提示用户并继续原始 prompt（不阻塞）
摘要任务：使用当前 CLI（更符合用户选择的 CLI 与模型能力预期）

- 摘要内容展示格式（适配当前 Webview 渲染）：
  - system 消息目前不渲染 Markdown，trace 为等宽逐行展示；因此摘要建议用“纯文本分段 + 每行前缀”。
  - 建议模板（示例）：
    - `【会话摘要】已压缩：<旧threadId> -> <新threadId>`
    - `FACTS:`（每行一个事实）
    - `TODOS:`（每行一个待办，前缀 [ ]/[x]）
    - `DECISIONS:`（每行一个决策）
    - `CONSTRAINTS:`（每行一个约束）
    - `INDEX:`（关键文件/命令/结论索引：每行 `type: value - note`）

### 压缩后的“可见性”和“可回滚”

由于选择了“新开 thread 继续、旧 thread 冻结”，天然支持回滚，但需要 UI 入口。

- 摘要展示：
  - A 作为 system 消息（可复制，用户易看到）
  - B 作为 trace 消息（更像过程日志，不打扰）
  - C system + trace（system 给结论，trace 留完整摘要）
  - 推荐：C（兼顾可见与不打扰）
- 回滚能力（是否做进 MVP）：
  - A MVP 不做回滚按钮，但保留冻结 threadId 在 metadata（人工可恢复）
  - B 做“撤销压缩/回到压缩前”按钮（面板）
  - 推荐：A（先把数据模型打好，后续补 UI）

### Runner 生命周期与上限（已确认）

交互模式下每个 CLI 仅维护 1 个“当前会话”的交互状态，但仍需定义释放策略，避免长期占用资源。

- 空闲释放（idle timeout）：
  - A 不做空闲释放（直到 VS Code 退出/切换会话）
  - B 24 小时无交互自动释放（已选）
  - C 30 分钟无交互自动释放
  - D 可配置（MVP 可先固定默认）
- 冻结 thread 保留数量（用于回滚/追溯）：
  - A 不限制（可能无限增长）
  - B 最多保留最近 5 个（已选）
  - C 最多保留最近 10 个

### 压缩后保留原文上下文（已确认）

因为选择“新开 thread 继续”，需要决定新 thread 的启动上下文里是否附带“最近 N 轮原文”，以保留短期细节。

- 选项：
  - A 只带摘要（最省 token，但可能丢短期细节）
  - B 摘要 + 最近 3 轮原文（已选）
  - C 摘要 + 最近 5 轮原文
  - D 可配置

### “停止运行”在交互模式下的语义（已确认）

SDK 交互模式下停止可能会影响 thread/runner 的一致性，需要明确用户预期与实现策略。

- 选项：
  - A 只停止当前 turn，runner 保持可继续（若 SDK 支持取消/中断）
  - B 停止当前 turn 并重建 runner（保留 threadId，下一次自动 resume；已选）
  - C 停止即关闭会话 runner，要求用户重新发送

## 已确认的实现细节（归档）

- Runner 生命周期与上限：
- 空闲释放：24 小时无交互自动释放
  - 冻结 thread 保留：最多保留最近 5 个
- 压缩后启动上下文：摘要 + 最近 3 轮原文
- 停止运行：停止当前 turn 并重建 runner（保留 threadId，下一次自动 resume）
